### REFACTOR BLUEPRINT REPORTED:

### Location
Found in `ui.js:29-46`, the `setActiveTab` and `setInactiveTab` functions.

### Description
These functions check `btn.tagName === 'MDUI-BUTTON'` to decide whether to set a CSS class or a `variant` property. This creates tight coupling between the UI abstraction layer and specific HTML element types. Adding any new button component (e.g., a custom web component) would require modifying these functions.

This is an anti-pattern because:
1. The UI layer shouldn't need to know about specific element implementations
2. It violates the Open/Closed Principle — open for extension, closed for modification
3. The logic is duplicated across both functions

### Suggested Refactor
Use a strategy pattern or duck-typing approach:
```javascript
function setActiveTab(btn) {
  if ('variant' in btn) {
    btn.variant = 'filled';  // MDUI-compatible components
  } else {
    btn.className = BTN_ACTIVE_CLASS;  // Standard HTML buttons
  }
}

function setInactiveTab(btn) {
  if ('variant' in btn) {
    btn.variant = 'text';
  } else {
    btn.className = BTN_INACTIVE_CLASS;
  }
}
```
Or better yet, define a `TabButton` interface/contract that both button types implement.
