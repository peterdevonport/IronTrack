### Description
`openProfileModal()` calls `history.pushState()` to add a modal entry to browser history, but `closeProfileModal()\) does not call `history.back()` or `history.go(-1)`. Instead, the back-navigation is handled entirely by the `popstate` listener in `app.js`. This asymmetry means the close path is non-obvious and fragile: clicking the modal's close button does NOT trigger a popstate event (because popstate only fires on actual browser back/forward), leaving the stale history entry in place. The next back press will pop the stale entry and re-run the popstate handler logic.

### Suggested Refactor
Either: (1) Have `closeProfileModal()` call `history.back()` to pop the entry, and let the popstate handler detect and skip the duplicate close, OR (2) Remove the `history.pushState` from `openProfileModal()` and handle modal state purely via DOM/state without polluting browser history. Option (2) is generally preferred since modals are transient UI state, not navigable routes.
