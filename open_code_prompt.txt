### Location

### Description
The light-mode badge overrides use CSS-escaped selectors to intercept and override Tailwind utility classes:

```css
.bg-purple-950\/50 { background-color: rgba(168, 85, 247, 0.12) !important; }
.text-purple-400 { color: #7c3aed !important; }
.border-purple-800\/60 { border-color: rgba(168, 85, 247, 0.3) !important; }
```

This approach has several problems:
1. **Fragile escape syntax** — The `\/ ` CSS escape is unusual and hard to read; editors and linters may not handle it correctly.
2. **Selector specificity escalation** — Each override requires `!important` to beat Tailwind's own specificity. This creates a specificity arms race where future Tailwind additions may need even higher specificity overrides.
3. **Breaks Tailwind's design contract** — Tailwind utility classes are meant to be composable and predictable. Intercepting them via CSS selectors creates hidden behavior that contradicts what the HTML says.
4. **Maintenance burden** — If Tailwind changes its class naming or if new badge variants are added, these overrides must be manually updated.

### Suggested Refactor
Follow the same pattern already established by `--shadow-card` — define semantic CSS custom properties for badge colors and reference them in the HTML:

```css
:root {
  --badge-pb-bg: rgba(168, 85, 247, 0.12);
  --badge-pb-text: #a855f7;
  --badge-pb-border: rgba(168, 85, 247, 0.3);
  --badge-1rm-bg: rgba(16, 185, 129, 0.12);
  --badge-1rm-text: var(--color-brand);
  --badge-1rm-border: rgba(16, 185, 129, 0.3);
}

.mdui-theme-light, .mdui-theme-auto {
  --badge-pb-text: #7c3aed;
  /* ... */
}
```

Then in the template HTML, replace Tailwind classes with inline styles or custom utility classes that reference these variables. This keeps the theme logic in CSS where it belongs and eliminates the Tailwind override pattern.
