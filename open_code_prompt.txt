### REFACTOR BLUEPRINT REPORTED:

### Location
Found in `plans.js:551`, inside the `listenToPlans` function.

### Description
When the plans filter is set to `'favorites'`, the snapshot listener calls `renderSharedPlansUI()` instead of passing the filtered list to `renderPlansUI()`. The favorites filter should display the user's own favorited plans, not shared plans. This causes the favorites view to never update reactively when plan data changes — it only updates via the initial `switchPlansFilter('favorites')` call.

```javascript
// Current (broken)
if (state.ui.plansFilter === 'favorites') renderSharedPlansUI();

// Should be
if (state.ui.plansFilter === 'favorites') {
  const favs = state.data.lastWorkoutPlans.filter(p => p.favorite);
  renderPlansUI(favs);
}
```

### Suggested Refactor
Extract the favorites filtering into a reusable helper, and call the correct render function from both `listenToPlans` and `switchPlansFilter`:

```javascript
function getFavoritePlans() {
  return state.data.lastWorkoutPlans.filter(p => p.favorite);
}

// In listenToPlans:
if (state.ui.plansFilter === 'favorites') renderPlansUI(getFavoritePlans());

// In switchPlansFilter 'favorites' branch:
renderPlansUI(getFavoritePlans());
```
This eliminates the duplicated filtering logic and fixes the reactive update bug.
